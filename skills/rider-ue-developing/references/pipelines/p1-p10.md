Follow these step-for-step. Do not invent shortcuts.

## P1. C++ edit → Live Coding → verify in PIE

> Domain decisions (which UCLASS/UPROPERTY conventions, GAS/Input/Network concern, modular pattern) belong to the **Architecture** and **C++ Code** references. This pipeline covers only the MCP plumbing.

1. `ue_status` — confirm editor connected.
2. `read_file` / `search_symbol` to locate target.
3. `apply_patch` to edit. (`replace_text_in_file` does not exist in this MCP version.)
4. `get_file_problems` on the edited file — fix everything red before building.
5. `build_solution_start` → `sessionId`.
6. Poll `build_solution_state(sessionId)` until `state != "Running"`. **If `buildIsSuccess == false`, stop here** — do not proceed to Live Coding; fix compiler errors and re-run from step 5.
7. `ue_get_logs(category="LogLiveCoding", pattern="patched|failed", count=20)` — confirm `Code successfully patched`.
8. If PIE was `Idle`: `ue_play(action="play", mode="floating", players=1)`. Wait 5–10 s; verify via `ue_status`.
9. `ue_get_logs(category="LogTemp", count=100)` — verify gameplay output.
10. `ue_play(action="stop")`.

> Live Coding compiles in new `UCLASS`/`USTRUCT`/`UENUM` (new files) and new `UPROPERTY`/`UFUNCTION` on existing types (the latter with a *"data type changes…"* warning) — the new type is usable in the running editor with **no restart**. It only rejects layout/signature changes to *existing* reflected types: changed parent, reordered/removed `UPROPERTY`, or changed reflected method signature. On an actual rejection: exit editor, `build_solution_start(rebuild=true)`, relaunch via `execute_run_configuration`.

## P2. Discover Blueprints derived from a C++ class

1. `get_class_hierarchy { baseClass: "WeaponInstance", limit: 5000 }` — backend walks full inheritance closure including abstract / `NotBlueprintable` bases.
2. Alternative: `search_assets { baseClass: "WeaponInstance" }` — same traversal + optional name filter.
3. `get_asset_properties { assetPath: "D:/abs/.../BP_Foo.uasset" }` on selected paths — returns Blueprint-level overrides only. Empty result (`[]`) means no overrides; read the C++ header for defaults (see **P8**).
4. **One field across the whole hierarchy:** `find_default_value_overrides { className: "WeaponInstance", fieldName: "SomeField" }` — returns every BP whose CDO overrides that UPROPERTY. `className` is the C++ base, not a Blueprint name.

## P3. Audit / refactor a GameplayTag

> Tag naming, hierarchy depth, "state tag vs event tag", redirector rules → see **Architecture** reference. This is the mechanics pipeline only.

1. `search_tags { prefix: "Ability.Damage" }` — enumerate existing tags.
2. `search_text` for the tag's literal in `.cpp`/`.h`.
3. Edit the C++ tag table; `apply_patch`.
4. `build_solution_start` and poll.
5. `ue_get_logs(category="LogGameplayTags", minVerbosity="Warning")` — confirm no unresolved tag warnings.

## P4. Crash / nullptr investigation

> Interpretation (GC reclaimed an unmanaged `UObject*`, `GetWorld()` in CDO construction, RPC reliability buffer overflow) → see **Architecture** reference.

1. `ue_get_logs(minVerbosity="Error", count=500)` — pull actual crash output.
2. `xdebug_get_debugger_status` — if session attached, dump it; else start one.
3. `xdebug_start_debugger_session` with editor's run configuration (`get_run_configurations` to find it).
4. `xdebug_set_breakpoint` on suspect file:line.
5. Reproduce: `ue_play(action="play", mode="floating", players=1)`.
6. On hit: `xdebug_get_stack`, `xdebug_get_frame_values`, `xdebug_evaluate_expression`.
7. `xdebug_set_variable` to test a fix hypothesis without rebuilding.
8. `xdebug_control_session("resume")` or `stop` to detach.

## P5. Editor automation via Python (single-shot)

1. `ue_status` — must be `connected`.
2. `ue_execute_python { script: "import unreal; ..." }`. (`isolated` is not a valid parameter — use `script` for a single snippet or `scripts` + `startFrom` for multi-step; see **P6**.)
3. Output capped at 10k chars — write large dumps to `Saved/*.txt` and `read_file` them.

> **Pitfalls — Python during PIE:**
>
> - **Never call `<SomeClass>.get_default_object()` on world-subsystem classes during PIE.** Example: `unreal.NavigationSystemV1.get_default_object()` triggers `Ensure condition failed: false || !HasAnyFlags(RF_ClassDefaultObject)` (NavigationSystem.h:297) and breaks into the debugger. To get the live NavigationSystem use `unreal.NavigationSystemV1.get_navigation_system(world)` where `world` is the PIE game world from `UnrealEditorSubsystem.get_game_world()`.
> - **Never pass a CDO as a world context.** `get_default_object()` returns the Class Default Object, not a `UWorld`. Passing it as `world` to any function that calls `UEngine::GetWorldFromContextObject` triggers a null-world warning and usually a crash.
> - **`PostExecGarbageCollect` / `CollectGarbage` during PIE can trigger `ForceDeleteObjects` ensures.** The fixed `PythonExecutor.cpp` defers GC via `GEngine->ForceGarbageCollection(false)` when `GWorld->IsPlayInEditor()`. If you see `Ensure condition failed: false [ObjectTools.cpp:3963] Failed to unload all packages during ForceDeleteObjects` pointing to a PIE-held asset (e.g. a Blackboard or DataAsset), this is the cause.

## P6. Multi-step content migration (resumable)

1. Build the script list as independently re-runnable Python snippets.
2. `ue_execute_python { scripts: [...], startFrom: 0 }`.
3. On failure: response contains `lastSuccessfulIndex`. Fix the offender.
4. Re-call with `startFrom: lastSuccessfulIndex + 1` — never replay completed steps.

## P7. PIE networking repro

1. `ue_play(action="play", mode="floating", players=2, netMode="listen", runUnderOneProcess=true, spawnAtPlayerStart=true)` — server in window #1, client in window #2.
2. **Verify the handshake landed** (ini showing `PIE_ListenServer` is necessary but not sufficient):
   ```
   ue_get_logs(sinceTimestampMs=t0, category="LogNet",
               pattern="listening on port|AddClientConnection|SendInitialJoin|Browse:",
               count=100)
   ```
   Require `"IpNetDriver listening on port"` server-side and `"SendInitialJoin"` client-side. If `Browse:` shows no `?listen` suffix, the game's `UGameInstance` is overriding standard PIE URL routing — see **pie-tools.md → Game-project UGameInstance override**.
3. `ue_get_logs(category="LogNet", minVerbosity="Warning", count=200)` for replication issues. (`follow` and `followTimeoutMs` are not valid `ue_get_logs` params — for continuous streaming use the **Monitor** tool polling `ue_get_logs` in a loop.)
4. `ue_play(action="pause")` + `ue_play(action="frame_skip")` to single-step a desync.
5. `ue_play(action="stop")` when done.

## P8. Find all Blueprints derived from a C++ class and inspect their values

> Use this pipeline for: "find blueprints that use `<Class>`", "show BP property values", "what does BP_X set for field Y".

1. `search_assets { baseClass: "MyClass" }` — returns every Blueprint `.uasset` that inherits from the C++ class (full transitive closure). Note the absolute path(s).
2. `get_asset_properties { assetPath: "D:/abs/path/.../BP_Foo.uasset" }` — Blueprint-level overrides only.
   - **Non-empty →** those are the BP overrides; C++ defaults fill the rest (step 3).
   - **`[]` →** no Blueprint-level overrides; all values equal the C++ defaults. Proceed to step 3.
3. **Read C++ defaults (no editor needed):** `search_text { q: "MyClass" }` → find the `.h` path in results → `read_file { file_path: "D:/abs/path/MyClass.h" }`. UPROPERTY in-class initialisers (e.g. `float WalkSpeed = 250.0f;`) are the ground truth. Collect the full property list here — you will need the exact names for step 4.
4. **Confirm live values via Python** (requires editor connected; use when you need certainty beyond static defaults):
   ```python
   import unreal
   bp  = unreal.load_asset('/Game/Relative/Content/Path/BP_Foo')
   cdo = unreal.get_default_object(bp.generated_class())
   # Use PascalCase names exactly as declared in the .h
   props = ['WalkSpeed', 'SprintTime', 'SprintSpeed']
   [print(p, '=', cdo.get_editor_property(p)) for p in props]
   ```
   **Do not call `dir(cdo)` to discover property names** — derive them from the header in step 3. Property names are PascalCase (C++ declaration), not snake_case.
5. **Audit one field across the whole hierarchy (optional):** `find_default_value_overrides { className: "MyClass", fieldName: "WalkSpeed" }` — returns every BP in the project that overrides this UPROPERTY. `className` is the C++ base, not a BP name. Requires both `className` and `fieldName`.

## P8b. Verify bot / AI behaviour in PIE (simulate_input + Slate tick-driver)

> Use when you need to confirm a gameplay variable (e.g. `GroundSpeed`) is non-zero while an AI pawn is actively chasing.

**Prerequisites — always check first:**
1. Confirm distance < AI's follow radius via Python *before* expecting movement. A bot at 643 units with `FollowRadius=300` will never chase — check positions first.
2. Use `simulate_input`, **not** `player.set_actor_location(...)`, to move the player. Python teleportation bypasses physics and produces unreliable bot reactions.

**Pipeline:**
1. Start PIE — `ue_play(action="play", mode="floating", players=1)`.
2. Get positions: `bot.get_actor_location()` / `player.get_actor_location()`. Compute distance; must be < `FollowRadius` and > `AcceptanceRadius`.
3. Calculate required `primitiveDuration` **before** arming: `duration = distance_to_cover / max_walk_speed`. For DemoPro57 (490 UU/s effective): 200 units → 0.41 s. An oversized duration blows the player past the follow radius and the bot stops.
4. Arm input ticker: `simulate_input { mode:"primitive", primitiveCall:"add_movement_input", primitiveDirection:"world_vec", primitiveWorldVec:[dx,dy,0], primitiveDuration:N }`. Returns immediately — player has NOT moved yet at this point.
5. Use the **Slate tick-driver** to sample state after movement is underway:
   ```python
   import unreal, math
   for k in ['_gs_h','_gs_s']:
       h = globals().get(k);
       if k.endswith('_h') and h: unreal.unregister_slate_post_tick_callback(h)
       globals().pop(k, None)
   world = unreal.get_editor_subsystem(unreal.UnrealEditorSubsystem).get_game_world()
   chars = unreal.GameplayStatics.get_all_actors_of_class(world, unreal.Character)
   bot = next((a for a in chars if 'Bot' in a.get_name()), None)
   comps = bot.get_components_by_class(unreal.SkeletalMeshComponent)
   s = {'t':0.0,'n':0,'h':None}
   def tick(dt):
       s['t']+=dt
       if s['n']>=5 or s['t']>10: unreal.unregister_slate_post_tick_callback(s['h']); return
       anim=comps[0].get_anim_instance(); vel=bot.get_velocity()
       sp=math.sqrt(vel.x**2+vel.y**2); gs=anim.get_editor_property('GroundSpeed')
       unreal.log(f'[GS] t={s["t"]:.2f} vel={sp:.1f} GroundSpeed={gs:.1f}')
       s['n']+=1
   s['h']=unreal.register_slate_post_tick_callback(tick)
   globals()['_gs_h']=s['h']; globals()['_gs_s']=s
   ```
6. After ~2 s: `ue_get_logs(pattern="[GS]", count=10)` — read the sampled values.
7. `ue_play(action="stop")`.

> **Pitfalls:**
> - `simulate_input` returns immediately — the player hasn't moved at poll time. Never call `ue_execute_python` to poll state right after arming; use the tick-driver callback instead.
> - Per-frame breakpoints on `Tick` pause the game every frame after resume. Disable the breakpoint after the first hit or use a conditional breakpoint (e.g. `Speed > 0`).
> - In Development builds the compiler inlines short-lived locals — `evaluate_expression` may report `'this' is not available`. Use `step_over` to materialise locals; for full visibility rebuild in Debug.
> - `GroundSpeed = 0` with `BotVelocity2D = 0` means the bot is NOT moving (NavMesh or distance issue), not that the GroundSpeed write is broken. Confirm the bot is actually moving before diagnosing the anim variable.

## P10. Scripted viewport scenarios — camera + spawn loop

Compose `viewport_camera` with `spawn_actor` to populate the level programmatically while keeping the camera framed.

1. **Pulse and reset.** `ue_status`, then `viewport_camera --action set --location [...] --rotation [...]` to baseline.
2. **Per iteration:**
   1. `viewport_camera --action get` — read current `location` + `rotation`.
   2. Compute world-space spawn point: `loc + forward(rot) * SPAWN_DISTANCE_CM`. UE forward vector for `(pitch,yaw,roll)`: `(cos(p)*cos(y), cos(p)*sin(y), sin(p))`.
   3. `spawn_actor --assetPath "/Engine/BasicShapes/Cube.Cube" --location [LX,LY,LZ] --label LABEL`.
   4. `viewport_camera --action move --delta [-RETREAT_CM,0,0] --relative true` — pull back along forward.
3. **Screenshot (optional):** `take_screenshot --kind viewport` between step 2.iii and 2.iv.
4. **Cleanup:** sweep actors by label prefix via `ue_execute_python` (see **scene.md** cleanup script).

Reference scenario: save a working script to `<project>/Saved/Scripts/scenario_spawn_and_retreat.py` for reuse.

> **Camera and player pawn are independent** — `viewport_camera` never touches PIE. To move the player view in PIE, drive via `simulate_input` or `ue_execute_python`.
